목표: Git 설치 → 저장소(repo) 연결 → Commit/Push → GitHub Pages로 웹 공개

수업 팁
학생들은 “내 화면이 강사 화면과 같은가?”를 확인하며 따라 합니다. 그래서 이 문서는 각 단계에 스크린샷(figure) + 설명 형태로 구성했습니다.

1) 핵심 개념 5개

  • Working tree: 지금 내 PC 폴더(파일을 편집하는 현장)
  • Stage: “이번에 올릴 변경분을 고르는 바구니”
  • Commit: 변경분에 “설명 라벨”을 붙여 하나의 기록으로 고정
  • Push: Commit 기록을 GitHub 저장소로 업로드
  • Publish(GitHub Pages): 저장소 내용을 웹사이트로 공개
중요
“저장(Save)”는 내 PC에만 저장입니다. GitHub에 반영되려면 Commit + Push가 필요합니다.

3) Part A. Visual Studio ↔ GitHub 연결(개요)

Visual Studio(IDE)는 자체적으로 Git 기능(UI)을 내장하고 있어 GitHub 로그인/클론/푸시까지 가능합니다. 다만 내부적으로는 Git 동작(또는 Git 구성 요소)이 필요하며, 환경에 따라 Git for Windows를 깔아두면 호환성이 가장 좋습니다.

왜 Visual Studio 연결과 VS Code 연결이 “새로 연결한 느낌”이었나?

  • 둘은 도구가 다르기 때문에 UI/로그인 흐름이 별개로 느껴집니다.
  • 하지만 “연결 대상”은 동일한 GitHub 저장소(repo)일 수 있습니다.
  • 즉, 저장소는 하나인데, 접속하는 편집기가 2개인 상황입니다.
정리
Visual Studio에서 연결했다고 해서 VS Code가 자동으로 따라오진 않습니다. 그래서 주인님이 느끼신 “두 번 연결” 감각은 정상입니다.

5) Part C. VS Code ↔ GitHub 저장소 연결 & 동기화

VS Code는 “편집기”이고, Git은 “버전관리 엔진”, GitHub는 “원격 저장소”입니다. VS Code에서 파일을 수정해도, GitHub에 올라가는 건 아닙니다. Commit + Push가 있어야 반영됩니다.

5-1. Welcome 화면은 닫아도 되는가?

5-2. .vs 폴더가 Changes로 올라오는 이유와 해결

  • .vs는 Visual Studio가 만드는 “로컬 설정/캐시” 성격의 파일이라 보통 저장소에 올리지 않습니다.
  • 그래서 .gitignore에 .vs/를 넣고, 이미 추적되던 파일은 git rm --cached로 추적에서 제거합니다.
# 1) 이미 추적되는 .vs를 "저장소 추적에서만" 제거 (PC에서는 남아도 됨)
git rm -r --cached .vs

# 2) 커밋
git commit -m "chore: remove .vs from repository"
주인님 화면에서 보인 “4 files changed, deletions(-)”
이건 “실제로 파일을 PC에서 삭제했다”가 아니라, GitHub에 올리는 추적 목록에서 제외했다는 의미가 핵심입니다.

6) Part D. “깨끗한 상태(Changes=0)” 만들기

정상 종료 조건

  • VS Code Source Control에 Changes 목록이 비어 있음
  • 상태 표시줄에 “Sync Changes”가 더 이상 쌓여 있지 않음
  • 터미널에서 git status 결과가 “working tree clean”에 가까운 상태

Sync Changes 1↑ 이 의미

  • 1↑는 “내 PC에 있는 커밋 1개가 아직 GitHub(origin/main)로 안 올라갔다(푸시 필요)”는 뜻입니다.
  • 이 상태에서 Sync Changes 또는 git push를 하면 원격 저장소와 동기화됩니다.
실전 팁
수업에서 가장 많이 실수하는 지점은 “Commit만 하고 Push를 안 하는 것”입니다. 그래서 “Sync Changes 숫자(↑)”를 확인 습관으로 만드세요.

7) Part E. GitHub Pages로 “이 가이드”를 웹사이트로 공개

GitHub Pages는 저장소의 특정 폴더(/docs) 또는 특정 branch를 웹으로 공개해 줍니다. 이번 실습에서는 /docs 폴더 방식이 가장 단순합니다.

7-1. 파일을 저장소에 넣기(로컬)

  1. 저장소 루트에 docs 폴더 생성
  2. docs/index.html에 이 파일 저장
  3. docs/assets/ 폴더 만들고, 캡처 이미지 파일을 규칙대로 넣기

7-2. Commit & Push

git add docs
git commit -m "docs: add VS & VS Code GitHub connection guide"
git push

7-3. GitHub Pages 설정(웹에서)

  1. GitHub 저장소 접속 → Settings
  2. 왼쪽 메뉴에서 Pages
  3. Build and deployment에서 Source를 선택:
    • Deploy from a branch
    • Branch: main, Folder: /docs
  4. Save를 누르면 잠시 후 공개 URL이 표시됩니다.
공개 URL 예시
보통 https://(GitHub아이디).github.io/(repo이름)/ 형태로 열립니다. (repo에 docs/index.html이 있기 때문에 루트로 접속하면 index.html이 자동 로드)

오해 & 교정

  • 오해: “VS Code에서 저장하면 자동 업로드된다” → 교정: 저장은 로컬, 업로드는 Commit+Push
  • 오해: “Source Control 숫자는 branch 개수” → 교정: 변경 파일 개수(아직 커밋 전)
  • 오해: “Sync는 위험하니 누르면 안 된다” → 교정: 동기화가 목적이면 정상 동작(단, 충돌은 설명 필요)

3-line summary

1) Git은 “엔진”, VS Code/Visual Studio는 “조종석”, GitHub는 “원격 창고”입니다.
2) GitHub 반영은 저장이 아니라 Commit + Push로 이루어집니다.
3) /docs에 index.html을 두고 GitHub Pages를 켜면, 저장소가 곧 웹사이트가 됩니다.
